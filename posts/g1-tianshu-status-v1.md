# G1产研相关项目现状总览（v1）

## 1) G1 product（地图讲故事原型）
### 当前状态
- 已有可运行原型，“地图讲故事”story 命中度高，演示效果不错
- 由产品经理 + Claude Code 快速迭代形成

### 主要问题
- 前端结构偏“堆叠式”：PC版、移动版是两个巨型页面
- 可维护性和可扩展性不足，研发规范薄弱

### 建议定位
- 保留为“产品体验基线仓”
- 逐步拆成可复用模块，不再继续无边界堆功能

## 9) 一体机项目（张军负责）
### 当前状态
- 已明确由张军牵头推进一体机方案
- 目标是将核心能力做成可交付的一体化形态

### 建设目标
- 打包集成：丰图地图 + AI问图 + OpenClaw 智能体能力
- 形成开箱可用、可演示、可部署的统一产品形态

### 关键价值
- 降低部署复杂度，减少对个人电脑和分散环境的依赖
- 提升交付稳定性与标准化程度，便于对外试点与复制推广

### 建议推进重点
1. 明确一体机软硬件边界（系统、模型服务、地图服务、Agent runtime）
2. 先做最小可用打包版本（MVP）并完成端到端联调
3. 与天枢主线接口保持一致，避免形成孤立交付分支
4. 建立版本发布与升级机制（镜像/安装包/配置模板）

## 一、下一步工作思路（总盘）
把所有工作分成4层，避免互相踩踏：

1. L0 战略与标准层（你主导）
   - 统一目标、统一架构红线、统一命名和交付标准
2. L1 平台底座层（天枢/Agent）
   - OpenClaw + 内网模型 + 工具编排 + 权限审计
3. L2 业务能力层（数据工厂/图谱/地图能力）
   - 地址、搜索、时空图谱、case推理、数据治理
4. L3 场景产品层（G1/宝安/深圳公安演示）
   - 地图讲故事、警务场景原型、汇报演示闭环

## 二、工作分工建议（按角色）
1) 你（总负责人）
- 只抓三件事：目标优先级、跨组冲突、里程碑验收
- 每周只看两个核心指标：Story命中率（产品价值）、可部署性（工程价值）

2) 产品线（马泽华 + 林位备）
- 负责：场景拆解、原型、用户流程、演示脚本
- 交付物：场景卡 + 原型 + 验收标准
- 约束：不自定义底层协议，不绕过平台能力

3) 研发线（李泽轮 + 何明 + 殷奎）
- 李泽轮：Agent编排与对话框架收口
- 何明：吴江历史资产（表结构/case）抽取与映射
- 殷奎：昆山数据标注平台能力实现（按统一接口接入）
- 约束：必须走统一 Tool Schema 和事件日志规范

## 三、推进策略（兼顾并行、隔离、可控）
采用“双周冲刺 + 周中评审”：
- 每两周一个可演示里程碑（不是文档里程碑）
- 周中 30 分钟架构评审：只审接口变更和风险
- 每组可独立开发，但必须满足：
  1. 同一接口契约（API/Tool Schema）
  2. 同一数据字典
  3. 同一 UI 设计 token（风格统一）
  4. 同一日志与审计规范

## 四、代码仓统筹方案（重点）
采用“多仓 + 统一治理仓”：
- g1-product：场景前端与 story 体验
- tianshu-agent-lab：agent 内核与编排
- ftools：地图能力适配层
- zllipractice：实验验证（孵化后回灌）
- kunshan-xxx：昆山标注平台（独立节奏）

新增：
- ft-platform-governance：规则与集成（架构、契约、数据字典、RACI、版本矩阵）
- ft-integration-runtime：集成编排（compose、联调脚本、环境模板、演示脚本）
